2. What data we collect

2.1 Email address (personal data)

When you start a free trial or complete a purchase, we collect the email address you enter in the Extension. This is the only directly identifying personal data we collect.

2.2 Browser fingerprint hash (technical data)

When you submit a verification request, the Extension generates a hash by combining a canvas-rendering signature with your browser's User-Agent string, applying SHA-256, and truncating to 24 hexadecimal characters. The hash is a one-way derivation — it cannot be reversed to identify you or your device model. It is used solely for rate-limiting repeated verification attempts from the same device and for detecting abuse patterns.

2.3 Licence and payment data

When you purchase a lifetime licence, we collect (from Stripe) the Stripe Checkout session ID, Stripe Customer ID, the purchase amount, currency, and timestamp. We do NOT receive or store your full card number, CVV, or card expiry — Stripe handles these as an independent data controller under PCI-DSS.

2.4 Consent records (evidence data)

At trial start and at purchase, we record the timestamp at which you accepted our Terms of Sale and Privacy Policy, along with the policy version. This forms part of our evidence bundle for consent and is used for dispute resolution.

2.5 Server-side interaction logs

When the Extension communicates with our Cloudflare Worker API, we log the request path (e.g. /send-verification, /verify), timestamp, HTTP response code, and in some cases your IP address. These logs are used for rate limiting, debugging, and abuse prevention. They are retained for up to 30 days unless required for longer for an active dispute.

2.6 Anonymous usage analytics

The Extension sends anonymised event data to Google Analytics 4 via the Measurement Protocol. Events include: use_installed, use_updated, use_popup_open, use_toggle, use_profile_change, use_theme_change, use_welcome_cta, use_logout, use_site_block. None of these events contain email, IP, URL, or any other personal data. A pseudonymous client ID is generated locally to aggregate events by session — this ID is not linked to your identity.

3. What data we do NOT collect

The Extension applies visual filters at the browser compositor level. It does not read the content of any webpage you visit. The following categories of data are never collected, stored, or transmitted by us:

  • Browsing history or lists of URLs visited;
  • Content of pages, including text, forms, passwords, cookies, or local storage of any website;
  • Credit-card data (handled only by Stripe);
  • Geolocation beyond what Cloudflare infers from an IP address for routing;
  • Health, biometric, political, religious, sexual-orientation, or other special-category data under Article 9 UK/EU GDPR;
  • Personal communications (we do not read messages, emails, chat content, or DMs);
  • Contact lists, calendars, or device files;
  • Precise device identifiers beyond the SHA-256 fingerprint hash described in section 2.2.

7. Permissions requested by the Extension and why

The Extension requests the following browser permissions under Manifest V3. Each is justified below.

Permission What it enables Data accessed
activeTab Apply visual filters to the tab you are currently viewing. Access to the active tab only when you invoke the popup. No content is read or stored.
storage Persist your preferences (active profile, theme, intensity) across browser sessions. Local data stored on your device. Not transmitted to us.
scripting Inject our content script into tabs that were opened before the Extension was installed. The content script only applies CSS filters; it does not read page content.
tabs Read the hostname of the active tab for the per-site disable feature. Only the hostname is read and only when you interact with the blocklist feature. Hostname is stored locally on your device and is never transmitted to us.

The content script match pattern is <all_urls> because the Extension's purpose is to enhance any website you visit. However, the content script only ever applies visual filters; it does not read or transmit page content. Chrome Web Store internal pages (chromewebstore.google.com/*) are explicitly excluded from the match pattern.

11. How we protect your data

  • All transport is over HTTPS (TLS 1.2+; TLS 1.3 preferred);
  • Cloudflare KV storage is encrypted at rest;
  • Stripe webhooks are cryptographically signed with HMAC-SHA256 and rejected if more than 5 minutes old;
  • Magic-link tokens are single-use and expire after 30 minutes;
  • API secrets are stored in Cloudflare Worker Secrets, not in source code;
  • Rate limits apply at IP, email, and fingerprint level to prevent abuse;
  • Disposable and temporary-email domains are blocked from signing up;
  • Trial state is tracked server-side and cannot be extended by local-storage manipulation.

Full technical details are published at our GitHub repository (security architecture document) where independently verifiable.

18. Technical limitations of the Extension

The Extension applies visual filters to webpages using SVG and CSS at the browser compositor level. Some content types cannot be filtered due to browser-level restrictions. In these cases, the Extension has no access to the content at all — it simply cannot apply filters. Affected types include:

  • DRM-protected video streams (Netflix, Disney+, HBO Max, Apple TV+, Amazon Prime Video — all use Encrypted Media Extensions);
  • Pages with strict Content Security Policy headers preventing injected styles;
  • Sandboxed iframes and cross-origin resources;
  • Banking and payment portals that block browser extensions by design;
  • Browser-internal pages (chrome://, chrome-extension://, about:, view-source:);
  • The Chrome Web Store itself (explicitly excluded);
  • Native PDF viewer;
  • Pure canvas / WebGL applications.

This is a browser limitation, not an extension defect. We never attempt to bypass these restrictions.
